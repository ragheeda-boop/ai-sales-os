Enterprise File Governance & Sync Architecture

AI Sales OS — Production-Grade Multi-Source Synchronization System

📅 Generated: March 31, 2026 🔐 Version: 1.0 📊 Status: Active Production

1. Executive Architecture

−

The AI Sales OS Sync Architecture orchestrates file governance across three distributed sources: Local Folder (working environment), Google Drive (business hub), and GitHub (code repository). The system applies the principle of hybrid push-pull synchronization with a decentralized source-of-truth model.

The Three Pillars

Local Folder ↔ Sync Engine ↔ Google Drive ↔ Manifest DB ↔ GitHub

Key Architecture Principles

  • No Single Source of Truth: Truth is decentralized and file-type specific
  • Manifest-Driven: Unified manifest built from 3 scans, drives all sync decisions
  • Conflict-First: Detection before resolution; manual review for binary conflicts
  • Hash-Based Deduplication: Files identified by content hash, not path/name
  • Backup-Before-Destructive: Every risky operation preceded by backup snapshot
  • Audit Trail Complete: Every sync action logged with source, target, decision, result

Sync Flow Architecture

ℹ️
Flow: Local edits → staging → sync decision → push to Drive/GitHub based on file type. Sync engine polls all 3 sources, builds unified manifest, detects changes, resolves conflicts per rules, executes sync actions, generates audit trail.

Decentralized Source of Truth Model

Each file type has its own source of truth. This eliminates conflicts by design:

  • Code files (.py, .js): GitHub is primary. Local is working copy. Drive doesn't store code.
  • Business docs (.docx, .pptx, .xlsx): Drive is primary. Local is working copy. GitHub stores only small/critical docs.
  • Markdown (.md): GitHub is primary (version history). Drive is secondary (collaboration).
  • Secrets (.env): Local ONLY. Never sync to Drive or GitHub.

3. Sync Rules Table

−

Deterministic rules that govern sync behavior for each file type and condition.

Rule Execution Priority

Rules are evaluated in order: Security rules (SR-007) execute first, then file-specific rules, then protection rules last.

ℹ️
Rule Conflict Resolution: If multiple rules apply to the same file, the rule with highest priority (smallest ID) wins.

4. Conflict Resolution Table

−

Deterministic conflict resolution strategy for each conflict type and file category.

Conflict Type File Types Strategy Auto/Manual Priority
Both modified (text) .py, .md, .json Attempt git merge/diff. If clean → auto. If conflicts → manual Auto then Manual GitHub wins if unsure
Both modified (binary) .docx, .pptx, .xlsx Create conflict copies: `file_LOCAL` + `file_DRIVE`. Flag manual review Manual Drive wins (latest approved)

5. File Classification Table

−

Detailed classification rules for each of the major file categories.

Code Files (.py, .js, .ts, .sql)

  • Primary Source: GitHub
  • Versioning: Git (full history)
  • Sync to Drive: NO (except HTML dashboards)
  • Sync to GitHub: YES (required)
  • Conflict strategy: Git merge for text files
  • Naming: snake_case, no spaces

Documentation (.md, .txt, .docx, .pdf)

  • .md/.txt: GitHub primary, Drive secondary
  • .docx/.pdf: Drive primary, GitHub optional
  • Versioning: Git for .md, Drive version history for .docx/.pdf
  • Conflict strategy: Git merge for .md, manual for .docx/.pdf

Presentations (.pptx, .ppt, .key)

  • Primary Source: Google Drive
  • Versioning: Drive version history
  • Sync to GitHub: NO
  • Sync to Drive: YES (required)
  • Naming: AI_Sales_OS_[Type]_v[X.Y].[ext]

Spreadsheets (.csv, .xlsx, .xls, .json)

  • .csv/.json: GitHub (if < 50MB, non-sensitive)
  • .xlsx/.xls: Drive primary
  • .parquet: Drive or GitHub depending on size

Media & Archives

  • Images < 1MB: GitHub OK
  • Images > 1MB or videos: Drive only
  • Archives (.zip, .rar): Drive only, never GitHub
  • Brand assets: GitHub + Drive

Config & Secrets

  • .env: Local ONLY, NEVER sync
  • .env.example: GitHub (template)
  • .gitignore, requirements.txt: GitHub
  • Dockerfile: GitHub

9. Local Folder Policy

−

Local is the Working Environment

  • Daily edits: All changes happen here first
  • Before syncing: Review changes, classify files, run manifest comparison
  • Staging area: _staging/ folder for files being prepared
  • Temporary exports: _temp/ folder, auto-cleaned after 7 days
  • Backup snapshots: Weekly automatic via backup_manager.py

Pre-Sync Checklist

  • (1) No uncommitted secrets
  • (2) No large binaries staged for git
  • (3) Manifest up to date
  • (4) No unstaged changes before push

Delete Safety

NEVER delete locally without checking if file exists in other sources first. Use the manifest to verify before deleting.

Key Modules

  • sync_engine.py: Main orchestrator — runs full sync cycle. Calls other modules in sequence.
  • scan_local.py: Walk filesystem, compute SHA256 hashes, build file list. Returns dict of {path: hash, metadata}
  • scan_drive.py: Use Google Drive API to list all files in project folder. Compute MD5 from API response.
  • scan_github.py: Use git commands to list tracked files + commits. Get hash from git object store.
  • build_manifest.py: Merge 3 scans into unified manifest. Deduplicate by hash.
  • compare_manifests.py: Compare current vs previous manifest. Return change list (added, modified, deleted, renamed).
  • detect_conflicts.py: For each file, check if it's modified in multiple sources. Flag conflicts.
  • resolve_conflicts.py: Apply conflict resolution rules. Return auto-resolved + manual queue.
  • sync_to_*.py: Execute sync actions. Handle API errors, retries, backups.

12. Sync Engine Pseudocode

−

Main Sync Flow

def run_sync():
    # 1. Pre-flight checks
    verify_credentials(drive_api, github_token)
    create_backup_snapshot()
    
    # 2. Scan all sources
    local_files = scan_local(LOCAL_ROOT)
    drive_files = scan_drive(DRIVE_FOLDER_ID)
    github_files = scan_github(REPO_NAME, BRANCH)
    
    # 3. Build unified manifest
    manifest = build_manifest(local_files, drive_files, github_files)
    
    # 4. Load previous manifest for comparison
    prev_manifest = load_manifest("sync_manifest.json")
    
    # 5. Detect changes
    changes = compare_manifests(manifest, prev_manifest)
    
    # 6. Classify and route
    for change in changes:
        change.classification = classify_file(change)
        change.source_of_truth = get_source_of_truth(change.classification)
    
    # 7. Detect conflicts
    conflicts = detect_conflicts(changes)
    auto_resolved, manual_queue = resolve_conflicts(conflicts)
    
    # 8. Execute sync actions
    for action in auto_resolved:
        if action.target == "local":
            sync_to_local(action)
        elif action.target == "drive":
            sync_to_drive(action)
        elif action.target == "github":
            sync_to_github(action)
    
    # 9. Save manifest
    save_manifest(manifest)
    
    # 10. Generate report
    report = generate_sync_report(changes, conflicts, manual_queue)
    log_audit_trail(report)
    
    return report

def build_manifest(local, drive, github):
    manifest = {}
    all_hashes = set()
    
    # Collect all hashes
    all_hashes.update(local.keys())
    all_hashes.update(drive.keys())
    all_hashes.update(github.keys())
    
    # For each unique hash, create FileRecord
    for hash in all_hashes:
        local_info = local.get(hash, {exists: false})
        drive_info = drive.get(hash, {exists: false})
        github_info = github.get(hash, {exists: false})
        
        record = FileRecord(
            hash=hash,
            local=local_info,
            drive=drive_info,
            github=github_info,
            status=determine_status(local_info, drive_info, github_info)
        )
        
        manifest[hash] = record
    
    return manifest

def detect_conflicts(changes):
    conflicts = []
    
    for change in changes:
        local_exists = change.sources["local"]["exists"]
        drive_exists = change.sources["drive"]["exists"]
        github_exists = change.sources["github"]["exists"]
        
        local_hash = change.sources["local"].get("hash")
        drive_hash = change.sources["drive"].get("hash")
        github_hash = change.sources["github"].get("hash")
        
        # Conflict: file modified in 2+ sources with different hashes
        hashes = [local_hash, drive_hash, github_hash]
        unique_hashes = set(h for h in hashes if h is not None)
        
        if len(unique_hashes) > 1:
            conflict = Conflict(
                file=change,
                type="both_modified",
                sources_affected=count_true([local_exists, drive_exists, github_exists]),
                severity="warning"
            )
            conflicts.append(conflict)
    
    return conflicts

resolve_conflicts(conflicts)

def resolve_conflicts(conflicts):
    auto_resolved = []
    manual_queue = []
    
    for conflict in conflicts:
        file_type = conflict.file.classification
        
        # Text files: try git merge
        if file_type in ["code_python", "markdown", "json"]:
            try:
                merged = attempt_git_merge(conflict)
                auto_resolved.append(merged)
            except MergeConflictError:
                manual_queue.append(conflict)
        
        # Binary files: never auto-merge
        elif file_type in ["presentation", "document", "spreadsheet"]:
            manual_queue.append(conflict)
        
        # Deleted-in-one: move to trash
        elif conflict.type == "deleted_in_one":
            action = MoveToTrash(file=conflict.file, reason="safe_delete")
            auto_resolved.append(action)
        
        # Rename: auto-update path
        elif conflict.type == "renamed":
            action = UpdatePath(file=conflict.file)
            auto_resolved.append(action)
    
    return auto_resolved, manual_queue

13. Real-World Conflict Examples

−

Example 1: Python File Modified Locally, GitHub Newer

Scenario: You edit daily_sync.py locally at 10:15 AM. GitHub has a newer commit at 11:00 AM.

  • Detection: Local hash != GitHub hash. Local timestamp < GitHub timestamp.
  • Classification: Code file → GitHub is Source of Truth
  • Decision: git pull --rebase to get GitHub version, then re-apply local changes on top
  • Auto/Manual: Auto (git merge). If conflicts → manual
  • Logging: SYNC | daily_sync.py | LOCAL_BEHIND | git pull --rebase | auto_resolved

Example 2: PowerPoint in Drive and Local Only

Scenario: AI_Sales_OS_Presentation.pptx exists in Drive (modified 10:00) and Local (modified 09:00). Not in GitHub.

  • Detection: Manifest shows Drive exists, Local exists, GitHub doesn't.
  • Classification: Presentation → Drive is Source of Truth. Should NOT be in GitHub.
  • Decision: Drive version is newer → download to local. No GitHub action.
  • Auto/Manual: Auto
  • Logging: SYNC | presentation.pptx | DRIVE_NEWER | download_to_local | auto

Example 3: Word Doc Modified in Both Local and Drive

Scenario: AI_Sales_OS_Audit.docx modified locally 14:00, Drive 15:30. Hashes differ.

  • Detection: Binary file. Two different hashes in 2 sources.
  • Classification: .docx → Drive is SoT. Both modified = CONFLICT
  • Decision: Cannot merge binary. Create conflict copies: Audit_LOCAL_20260331.docx + Audit_DRIVE_20260331.docx. Flag manual review.
  • Auto/Manual: Manual
  • Logging: CONFLICT | Audit.docx | BOTH_MODIFIED_BINARY | conflict_copies_created | manual_review

Example 4: Excel File Moved Between Folders

Scenario: companies.xlsx was in data/imports/. Now in data/exports/. Same size, same hash.

  • Detection: Hash matches file in old location. New path detected.
  • Decision: Treat as move/rename. Update manifest path. Propagate move to Drive.
  • Auto/Manual: Auto
  • Logging: MOVE | companies.xlsx | data/imports/ → data/exports/ | path_updated | auto

Example 5: PDF Duplicate Detected

Scenario: Two PDFs with same hash but different names: report_final.pdf and report_v3.pdf

  • Detection: Hash matching identifies duplicates.
  • Decision: Flag as duplicate. Keep the one in correct folder per governance. Archive the other.
  • Auto/Manual: Manual (user decides which to keep)
  • Logging: DUPLICATE | report_final.pdf = report_v3.pdf | same_hash | manual_review

Example 6: .env Detected (Security Block)

Scenario: .env file found locally with API keys.

  • Detection: Filename matches secret pattern.
  • Classification: Secret file → Local ONLY
  • Decision: BLOCK sync. Verify .gitignore includes .env. Alert if staged for git.
  • Auto/Manual: Auto (block)
  • Logging: BLOCK | .env | SECRET_FILE | sync_blocked | security

  • Never delete audit trail (compliance requirement)

15. Safety Rules

−

These are non-negotiable rules that protect the integrity of the multi-source sync system.

⛔
Rule 1: NEVER delete a file permanently without creating a backup first
⛔
Rule 2: NEVER propagate a delete from one source to all sources automatically
⛔
Rule 3: NEVER sync .env, credentials, API keys, or secrets to GitHub or Drive
⛔
Rule 4: NEVER auto-merge binary files (.docx, .pptx, .xlsx, .pdf, images)
⚠️
Rule 5: ALWAYS create a backup snapshot before any destructive sync operation
⚠️
Rule 6: ALWAYS preserve conflict copies — never overwrite without backup
⚠️
Rule 7: ALWAYS verify .gitignore includes all sensitive patterns before git push
ℹ️
Rule 8: Any file missing from one source is NOT assumed deleted — flag for review
ℹ️
Rule 9: Any overwrite operation must be reversible within 30 days
ℹ️
Rule 10: Binary conflict → manual review queue, never auto-resolve
ℹ️
Rule 11: If sync fails mid-operation → rollback to last backup, log error
ℹ️
Rule 12: Maximum 3 auto-retries on API failures, then halt and alert
ℹ️
Rule 13: Sync engine must run health check before and after each cycle
ℹ️
Rule 14: All sync operations logged to audit trail — no silent operations
ℹ️
Rule 15: Weekly backup verification — ensure backups are complete and restorable

Enforcement

  • Rules 1-4 are code-level assertions. Violation = sync halts with CRITICAL error.
  • Rules 5-7 are pre-flight checks. Violation = warning logged, sync continues but with caution flag.
  • Rules 8-15 are operational policies. Violation = post-operation review required.

Emergency Override

Only Ragheed (owner) can override rules 1-4. All overrides must be logged with reason and timestamp. Overrides trigger mandatory post-mortem review.
