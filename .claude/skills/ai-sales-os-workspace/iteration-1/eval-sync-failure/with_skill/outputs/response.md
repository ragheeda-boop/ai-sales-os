# إصلاح خطأ 429 من Apollo وإعادة تشغيل Sync

## المشكلة: رسالة الخطأ 429

رسالة الخطأ **429 Too Many Requests** من Apollo تعني أنك تجاوزت حد معدل الطلبات (Rate Limit) المسموح به. Apollo لديها قيود على عدد الطلبات التي يمكنك إرسالها في فترة زمنية معينة.

## لماذا يحدث هذا؟

هناك عدة أسباب محتملة:

1. **عدد الطلبات كثير جداً:** إذا كان الـ sync يعمل مع عدد كبير من السجلات (44,875 contact و 15,407 company)، قد يرسل طلبات بسرعة أكبر من حد Apollo
2. **مشاكل الشبكة:** قد تكون هناك مشاكل مؤقتة في الاتصال بـ Apollo
3. **عمليات متزامنة:** إذا كان لديك عدة عمليات تعمل بنفس الوقت على Apollo API

## كيفية الإصلاح والتشغيل مرة أخرى

### الخطوة 1: فهم آلية إعادة المحاولة الحالية

كود `daily_sync.py` لديه بالفعل **5x exponential backoff** على أخطاء 429 و 500 في `notion_helpers.py`. هذا يعني:
- المحاولة الأولى: انتظر بعض الوقت
- المحاولة الثانية: انتظر أكثر (مضروب في 5)
- وهكذا حتى 5 مرات

إذا فشلت بعد 5 محاولات، فالمشكلة أكبر من خطأ مؤقت.

### الخطوة 2: تحديد نوع الـ Sync الذي فشل

افحص اللوج (Log) لترى أي وضع كان يعمل:
```
❌ إذا كان: "mode incremental" → المشكلة عادية (يومية)
❌ إذا كان: "mode backfill" → ربما حاولت استعادة بيانات قديمة
❌ إذا كان: "mode full" → ربما تفعل reload كامل
```

### الخطوة 3: انتظر قبل إعادة المحاولة

**أول شيء: لا تعيد محاولة فوراً!**

- انتظر **15-30 دقيقة** قبل إعادة التشغيل — هذا يعطي حد Apollo وقتاً لإعادة تعيين
- لا تشغل عدة syncs بنفس الوقت
- تأكد من عدم وجود عمليات أخرى تستخدم Apollo API

### الخطوة 4: إعادة التشغيل الصحيحة

اختر الأمر المناسب حسب الحالة:

#### إذا كان sync عادي (incremental):
```bash
cd "💻 CODE/Phase 3 - Sync"
python daily_sync.py --mode incremental --hours 26
```
هذا يسحب آخر 26 ساعة من البيانات. الـ 2 ساعة إضافية (26 بدل 24) توفر تداخل لحماية من فقدان البيانات.

#### إذا كان sync backfill (استعادة بيانات):
```bash
python daily_sync.py --mode backfill --days 7
```

**ملاحظة مهمة:** إذا تم الـ sync بشكل جزئي قبل الفشل، يوجد ملف `backfill_checkpoint.json` يحفظ نقطة التوقف. إعادة تشغيل نفس الأمر ستبدأ من حيث توقفت، وليس من البداية.

#### إذا كنت تريد إعادة بناء كاملة (Full):
```bash
python daily_sync.py --mode full
```
هذا **قد يستغرق 2-4 ساعات** وسيسحب **جميع** السجلات مع استخدام alphabetical partitioning (A-Z + numbers) للتعامل مع حد Apollo البالغ 50,000 سجل لكل بحث.

### الخطوة 5: مراقبة النتائج

بعد إعادة التشغيل مباشرة:

1. **تحقق من عدد السجلات:**
   ```
   ✅ Synced X companies
   ✅ Synced Y contacts
   ✅ No duplicates (أو أرقام منخفضة < 5%)
   ```

2. **تحقق من الأخطاء في اللوج:**
   - أي رسائل 429 جديدة؟
   - أي أخطاء في الاتصال بـ Notion؟

3. **تحقق من صحة البيانات:**
   إذا أردت التأكد من عدم وجود أخطاء عميقة، شغّل health check:
   ```bash
   python health_check.py
   ```

## إذا استمرت المشكلة (429 يتكرر)

إذا كان الخطأ 429 يستمر بعد الانتظار وإعادة المحاولة:

### السبب المحتمل 1: حد Apollo قليل جداً
- ربما خطتك على Apollo لا توفر معدل طلبات عالي كافي
- **الحل:** تواصل مع Apollo support أو ارقِ الخطة

### السبب المحتمل 2: المشاكل الشبكية المستمرة
- قد تكون مشاكل في الاتصال بين خادمك و Apollo
- **الحل:** تحقق من الاتصال، جرب VPN مختلف إن أمكن

### السبب المحتمل 3: الـ Sync يسحب بيانات كثيرة جداً مرة واحدة
- إذا كنت تستخدم `--mode full`، هذا يسحب 60,000+ سجل
- **الحل:** استخدم `--mode incremental` للعمل اليومي بدل `--mode full`

## معلومات مهمة عن الـ Sync

- **Triple Dedup:** الـ Sync يستخدم 3 طرق لمنع التكرار: Apollo ID → Email → memory set
- **Safe Boolean Writing:** لا يكتب engagement checkboxes إلا إذا أرسل Apollo البيانات (يحمي من الكتابة فوق البيانات اليدوية)
- **Seniority Normalization:** يوحّد "C-Suite"/"C suite" variants إلى صيغة موحدة
- **Pre-loading:** يحمّل جميع السجلات الموجودة من Notion أولاً لتقليل عدد الطلبات
